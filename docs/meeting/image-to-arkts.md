# Image-to-ArkTS 项目周会准备（第一周）

## 1. Coding Agent 与 Test Agent 的衔接与集成

### 1.1 当前进展
*   **独立测试阶段**：目前的 Test Agent 尚未集成到主流程中，仅能对 DevEco 项目进行单独的编译测试。
*   **输入与验证**：以整个 DevEco 项目为输入，验证其能否成功编译。若编译失败，则捕获并输出错误日志。

### 1.2 存在问题与待确认事项
*   **接口定义不明确**：
    *   已确认 Architect Node 输出 UI 转化的 JSON 数据。
    *   **关键缺失**：尚不明确 Coding Agent 的具体输出格式和内容（例如是完整项目代码还是代码片段）。
*   **适配需求**：由于输入输出标准未定，后续可能需要大幅调整 Test Agent 的输入处理逻辑，以适配 Coding Agent 的产物。

### 1.3 环境配置注意事项
*   **编译依赖**：运行测试前，必须在 local.properties 文件中显式配置 SDK 和 Node.js 的路径，否则会导致编译环境检查失败。

## 2. Test Agent 功能规划与演进

### 2.1 核心挑战：动态交互与功能验证
*   **引入 Computer Use Agent 的必要性**：
    *   **痛点**：目前的 Test Agent 难以直接验证动态交互逻辑（如按钮跳转、动画效果等）。
    *   **设想方案**：为减少人工干预，建议引入 **Computer Use Agent** 技术。通过自动截取运行时 UI 截图并提交给多模态 Agent，进行智能视觉比对和功能验收，以检测：
        1.  UI 渲染是否符合设计预期。
        2.  动态按键功能是否正确实现。
        3.  是否存在遗漏的交互元素。

*   **测试依据与功能文档**：
    *   **问题**：当前缺失结构化的功能描述文档，导致 Test Agent 无法进行有效的黑盒/功能测试。
    *   **解决方案**：建议在 **Architect Node** 阶段，将自然语言的需求转化为标准化的 **JSON Schema** 格式的功能规格说明书，作为 Test Agent 的测试依据（Oracle）。

### 2.2 版本定位与路线图
*   **v1.0 (当前版本)**：
    *   **定位**：本质上是一个用于 **Function Calling** 的工程化工具（Tool）。
    *   **职责**：紧接在 Coding Agent 之后，专注于代码的编译与构建测试。
    *   **局限**：尚未集成大模型（LLM）能力和 Prompt 工程，属于确定性的自动化脚本工具。
